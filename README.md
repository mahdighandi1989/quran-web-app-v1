# Quran Web App (React + Vite)

این پروژه نسخهٔ آمادهٔ اجرای کد شماست.

## اجرا (لوکال)
1) Node.js 18+ نصب باشد.
2) متغیرهای محیطی Firebase را تنظیم کنید (بخش «پیکربندی محیط» پایین).
3) در پوشهٔ پروژه دستور زیر را بزنید:
```bash
npm install
npm run dev
```
۴) آدرسی که Vite می‌دهد (مثلاً http://localhost:5173) را باز کنید.

## پیکربندی محیط (Firebase)
اپ یک پیکربندی پیش‌فرضِ **عمومیِ** Firebase داخل `src/lib/firebase.js` دارد، پس بدون هیچ
تنظیمی کار می‌کند. اگر می‌خواهید برای محیط خودتان آن را override کنید، متغیرهای محیطی
`VITE_*` را تنظیم کنید — این مقادیر بر پیش‌فرض **اولویت** دارند (Vite آن‌ها را در زمان build
تزریق می‌کند). کلید web فایربیس «سِکرت» نیست؛ امنیت با محدودسازی کلید + Authorized domains اعمال می‌شود.

1) (اختیاری) فایل نمونه را کپی کنید:
```bash
cp .env.example .env
```
2) در `.env` مقادیر پروژهٔ Firebase خود را قرار دهید
   (`VITE_FIREBASE_API_KEY`، `VITE_FIREBASE_AUTH_DOMAIN`، `VITE_FIREBASE_PROJECT_ID`،
   `VITE_FIREBASE_STORAGE_BUCKET`، `VITE_FIREBASE_MESSAGING_SENDER_ID`،
   `VITE_FIREBASE_APP_ID`، `VITE_FIREBASE_MEASUREMENT_ID`) — از
   Firebase console → Project settings → General → Your apps.
3) فایل `.env` در `.gitignore` است و **نباید** commit شود.
4) برای **استقرار (deploy)** در صورت تمایل همین متغیرها را در environment پلتفرم میزبان
   (Render/Vercel/Netlify) ست کنید؛ در غیر این صورت پیش‌فرضِ داخلی استفاده می‌شود.

### Inspector Bridge (ابزار دیباگ، اختیاری)
اسکریپت Inspector Bridge داخل `index.html` فقط زمانی فعال می‌شود که
`VITE_ENABLE_INSPECTOR_BRIDGE=true` باشد. در production (پیش‌فرض) کاملاً غیرفعال است و
هیچ اتصال WebSocket یا خطایی در کنسول ایجاد نمی‌کند. برای فعال‌سازی در توسعهٔ محلی، آن را
در `.env` روی `true` بگذارید (و در صورت نیاز `window.__INSPECTOR_WS_URL__` را پیش از بارگذاری ست کنید).

## تست
این پروژه از [Vitest](https://vitest.dev/) (همراه با React Testing Library و jsdom) برای تست استفاده می‌کند.

```bash
npm test          # اجرای یک‌بارهٔ همهٔ تست‌ها
npm run test:watch  # اجرای تست‌ها در حالت watch
```
- پیکربندی تست در `vitest.config.js` و تنظیمات global در `src/test/setup.js` قرار دارد.
- فایل‌های تست با الگوی `src/**/*.{test,spec}.{js,jsx}` شناسایی می‌شوند.

## تعامل و سنجش پذیرش (Engagement Analytics)
برای سنجش و افزایش **تعامل کاربر**، هر اقدام معنادار کاربر (باز شدن اپ، مشاهدهٔ تب،
کامل‌شدن جلسهٔ تمرین/آزمون، تعامل با onboarding) از طریق `src/lib/analytics.js`
ثبت می‌شود: هم به Firebase Analytics (رویداد سفارشی GA4) ارسال می‌شود و هم در یک
دفترچهٔ روزانهٔ محلی ذخیره می‌شود تا KPI اصلی — **«تعامل‌های یکتای روزانه» (هدف: ≥ ۵۰۰ در روز)** —
هم در production و هم در تست‌ها قابل اندازه‌گیری باشد.

- تعریف دقیق KPI، تحلیل شکاف، و معیارهای effectiveness در [`docs/engagement-kpi.md`](docs/engagement-kpi.md).
- `engagementReport()` وضعیت فعلی نسبت به هدف را برمی‌گرداند (`{ today, target, met, gapToTarget, days }`).
- مؤلفهٔ `EngagementNudge` کاربران کم‌فعالیت را با یک فراخوان روشن به تمرین، و با
  A/B test برای دو flow مختلف onboarding، به سمت ارزش اصلی اپ هدایت می‌کند.
- تست outcome به‌صورت end-to-end در `src/test/engagement.e2e.test.jsx` صحت رسیدن به هدف را می‌سنجد.

## تلگرام (اعلان‌ها و تعامل)
تب تلگرام **فقط پس از ورود (Google sign-in)** در **تنظیمات** نمایش داده می‌شود و پیکربندی آن
**برای هر حساب جداگانه در Firestore (بک‌اند)** ذخیره می‌شود — **هیچ‌چیز در حافظهٔ مرورگر**
نگه‌داری نمی‌شود (نه توکن، نه Chat ID). از همان‌جا می‌توانید:
- توکن بات و Chat ID را ثبت و **دستگاه‌های** دیگر را اضافه کنید.
- انتخاب کنید **کدام نوع اعلان** فعال و هرکدام **صدادار/بی‌صدا** باشد.
- **یادآوری** تعریف کنید و «پیام آزمایشی / ارسال وضعیت / تنظیم منوی بات» را بزنید.

ارسالِ اعلان‌ها **خروجی** است (از مرورگرِ کاربرِ واردشده به تلگرام). توکن سِکرت است و فقط در
Firestore و مخصوص حساب شما ذخیره می‌شود (و با قوانین `firestore.rules` فقط برای خودتان قابل‌خواندن است).

پیش‌نیاز یک‌بار: **Firestore را در پروژهٔ Firebase فعال** کنید و `firestore.rules` را منتشر کنید
(جزئیات در `TO-DO/`).

### کنترل دوطرفه (مشاهده و کنترل از داخل تلگرام)
این بخش پیاده‌سازی شده است: برنامه یک خلاصهٔ وضعیت را در `appState/{uid}` در Firestore
mirror می‌کند و سرور بات ([`server/`](./server/README.md)) با Firebase Admin همان داده را
می‌خواند/می‌نویسد. پس از اجرای سرور و تنظیم webhook:
- `/status`، `/progress`، `/today` دادهٔ واقعی همان حساب را برمی‌گردانند.
- `/remind HH:MM متن` یک یادآوری در Firestore ثبت می‌کند که برنامه آن را **به‌صورت realtime**
  دریافت می‌کند (onSnapshot) و هنگام باز بودن اجرا می‌کند.
- منوی ثابتِ پایین به همین دستورها نگاشت شده است.

اتصال حساب: در تب تلگرام، Chat ID خود را وارد کنید (دستور `/start` آن را نشان می‌دهد)؛ بات از
روی همین شناسه تشخیص می‌دهد کدام کاربر است. راه‌اندازی سرور و service account در `server/README.md`.

## هوش مصنوعی (کلیدها و مدل‌ها)
در **تنظیمات → بخش «🧠 هوش مصنوعی»** می‌توانید کلید مدل‌های هوش مصنوعی را مدیریت کنید:
- چند **پروایدر پیش‌فرض** (OpenAI، Anthropic/Claude، Google Gemini، OpenRouter، Groq، DeepSeek)
  هرکدام با فهرستی از **آخرین مدل‌ها**.
- **افزودن پروایدر سفارشی** (سازگار با OpenAI/Anthropic/Gemini) با Base URL و **اعتبارسنجی خودکار** کلید.
- **افزودن/مپینگ مدل جدید** برای هر پروایدر با **اعتبارسنجی** (یک فراخوانی کوچک تست).
- ذخیره‌سازی per-user:
  - وقتی **وارد حساب** هستید، کلیدها در Firestore (`aiConfigs/{uid}`) و **فقط برای خودتان** ذخیره و
    بین دستگاه‌ها همگام می‌شوند؛ پس از خروج دیده نمی‌شوند.
  - وقتی **مهمان** هستید (بدون ورود)، می‌توانید کلید خودتان را وارد و استفاده کنید، ولی کلید
    **فقط در همان نشست مرورگر** می‌ماند و **هرگز ذخیره نمی‌شود**.

⚠️ امنیت: فراخوانی مستقیم سرویس از مرورگر، کلید را در همان درخواست افشا می‌کند (مناسب «کلید شخصی
خودت»). برای کلید اشتراکی، از یک پراکسی بک‌اند استفاده کنید.

## نکات
- کتابخانهٔ XLSX به‌صورت داینامیک از CDN لود می‌شود.
- داده‌ها (sessions, settings, dataset) در LocalStorage مرورگر ذخیره می‌شود.
- برای پخش صوت، اتصال اینترنت لازم است.
- برای تشخیص گفتار (🎙️) مرورگر باید Web Speech API را پشتیبانی کند و زبان آن روی `ar-SA` ست شده است.
